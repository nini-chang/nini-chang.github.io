July 14, 2021 · 2 min · Nini Chang

物件導向 (Object-oriented Programming)#

先來談談物件導向是什麼

  • 被理解為一種將程式分解為「封裝資料」及相關操作的「模組」，而進行的程式設計
  • 通常共享高階程式的低階功能
  • 基本工具：variable, function
  • 利用繼承達到程式碼複用性、事物抽象化和物件的延伸性
  • 訊息傳遞：動態組態（方法隨實例化動態的變化）

JavaScript 的 OOP 是 prototype-based，可參考: JavaScript 類似繼承設計與原型鍊

OOP 三大特性：

  • 封裝 (Encapsulation)
  • 繼承 (Inheritance)
  • 多型 (Polymorphism)

封裝 (Encapsulation)#

目的是簡化、提升系統安全性。將物件內部具體實現的方法隱藏起來，使用者只能透過物件本身提供的介面（interface）取得物件內部屬性或方法。

通常會賦予 public, private, protected 限制使用的對象。Public 可以被所有人都使用，private 僅限類別內部使用（子類別無法直接存取），protected 表示同一 package 成員可以直接存取，其他 package 必須透過繼承的方式才可以在子類別中存取。

繼承 (Inheritance)#

類別物件的資源的「延伸」和「重複使用」。而延伸使用的稱為子類別，子類別會比父類別更加具體化。

class Bike {
  public String brand;
  public String Color;
  public int price;
  public String getBikeInfo() {
    return "Brand: " + brand + ", Color: " + color + ", Price: " + price;
  }
}

class Giant extends Bike {
  public String riderName;
}

多型 (Polymorphism)#

由繼承產生相關不同的 class，其物件會對同一訊息做不同反應。意即，多個相同名稱的方法，傳入不同參數會執行不同的敘述。其中包含多載（overloading）和覆寫（overriding）。

  • 多載：相同類別中，定義名稱相同，但參數個數不同，或參數型態不同的函式 -> 用此呼叫到對應的方法
public int area(int length) {
  return length * length;
}

public int area(int length, int width) {
  return length * width;
}
  • 覆寫：覆寫掉父類別中的函式。
class Animal {
  protected String sounds;
  protected virtual String getSounds() {
    return sounds = "hahaha";
  }
}

class Dog extends Animal {
  protected override String getSounds() {
    return sounds = "Bow!";
  }
}

物件導向的五個基本原則：SOLID#

  • Single responsibility principle (SRP)

    • 一個 class 只負責一件事情，越單純越好。如果負責太多任務，表示要重構

  • Open / Close Principle (OCP)

    • 開放（擴充）封閉（修改）原則
    • 耦合度(coupling)不可以太高，否則無論擴充或修改皆可能影響功能
    • 引述影片中範例：今天想要汽車可以在雪地上行走，「加裝雪鍊」在輪胎上（開放新增），但又「不改變輪胎原有的功能」（封閉修改

  • Liskov Substitution Principle (LSP)

    • 在一個系統中，child class 應該可以替換掉 parent class 而不會影響結構
    • 參照文章中範例 - 有三種車子：樂高車、超跑、露營車，樂高車是由積木組成的模型車、沒有引擎，根本不能上路，這時候子類別樂高車，並沒有辦法執行父類別車子的路上跑功能，這種情況就不符合 Liskov 替換原則（子類別應該可以執行父類別想做的事情）

  • Interface Segregation Principle (ISP)

    • 把不同功能從介面中分離出來
    • 延續上述例子來說：樂高車明顯違反 Liskov 原則，因此我們必須修改車子的定義，大部分的車子有「路上跑」功能，我們可以將這個功能分割到其他介面
    • 用戶不應該被迫相依於他們用不到的 function。如果有用不到的 function 或是 protocol，有要用到再裝上去就好

  • Dependency-Inversion Principle (DIP)

    • 定義：高階模組不應依賴低階模組，兩者應皆依賴抽象概念，抽象概念不依賴細節，而是細節依賴在抽象概念
    • 引述影片中範例：我原本會請王阿姨來打掃（耦合王阿姨），但如果今天王阿姨請假不來打掃，我的房間會髒亂，那麼依賴反轉就會是，我現在請清潔公司派人來打掃（耦合清潔公司），如果今天原本會來打掃的人不來了，清潔公司就會改派有空的人來打掃。
